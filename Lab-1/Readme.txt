1 Часть
-------------------
-------------------

Bad practice
1) Использование большого базового образа. Далеко не всем контейнерам нужен функционал большого базового образа. Можно скачать облегченную версию без потери функциональности, но с экономией веса. Без исправления контейнер начинает слишком много весить и долго загружаться
Исправление: использовать слим образы

2) создание дополнительных слоев в докер-образе при помощи лишних инструкций в докерфайле. Плохая практика т.к в UFS существуют ограничения на количество слоев в образе. Так, например в AUFS(на момент выхода книги) существует	ограничение в 127 слоев. Слои могут закончится.
Исправление: Объединение инструкций в 1

3)Опасное использование томов в докерфайле. При неудачном исходе злоумышленник может подключиться к докеру и воспользоваться уязвимостью хоста. Тем самым он получит доступ ко всей файловой системе(угроза усиливается запуском докера от имени root без использования namespaces).
Это может грозить дос атакой, воровством данных и т.д
Исправление: убрать волюм хост (+ настроить namespaces и cgroups)


4)Собирать образ без флага --no-cache если в нем присутствуют инструкции обновления.
При сборке образа демон докера кэширует каждый слой, чтобы ускорить сборку(пропуск уровней при одинаковом кэше). Если будет нужно пересобрать образ, то одинаковые инструкции пропустятся и автоматически перейдут в слой образа. Получается, если пользователь захочет пересобрать образ, чтобы обновить пакет приложений, у него ничего не выйдет. Чтобы каждая инструкция выполнялась заново, нужно добавить в docker build флаг --no-cache, тогда инструкция выполнится и приложения обновятся.

4.1) Если пользователь хочет пересобрать образ с обновленным базовым образом, то ему следует использовать флаг --pull (Docker build --pull -t imagename), так демон подтянет новый обновленный образ с докерхаба. Это происходит из-за того, что команда build берет в первую очередь локальный базовый образ, даже если он устарел

5) Не использовать .dockerignore. При неправильном размещении dockerfile демон докера начнет подгружать в образ много ненужной информации. Например, если разместить докерфайл в папке downloads, то демон будет подтягивать все загрузки. Это увеличивает размер контейнера. С использованием .dockerignore можно задать игнорируемые каталоги, которые не стоит упаковывать в образ. Таким образом контейнер будет без мусора/важных данных, которые не должны в него попадать.

2 часть
-------------------
-------------------

Bad practice
1) Использование томов. В плохом докер compose файле в сервисе web2 вообще нет тома, что значит, что изменения на сайте не сохраняются. В web3 должна быть один и тот же вид сайта, но том указан неправильно. Чтобы контейнер подгружал файлы с хранилища на локальном устройстве ему необходим правильно указанный том.
Исправление: добавить том с правильным путем. В исправленном файле вид веб3 сохраняется, веб2 записывает логи на локальный диск

2) При отсутствии Health check + depends on контейнер может начать подключаться к еще не инициализированной базе данных, что может его крашнуть. Сам Health check позволяет проверить запуск сервиса с нужными параметрами.
Исправление: использовать Health check. В исправленном yaml файле контейнеры ждут проверки бд до запуска.

3) Использование root при подключении к базе данных очень -- опасная практика. Злоумышленник или сотрудник обладает всеми правами изменения/удаления всех баз данных, что может повлечь за собой риск потери данных и сопутствующих убытков.
Исправление: Создать пользователя с ограниченными правами и выдать ему доступ к нужной бд. В исправленном файле настроен usr1 который пользуется бд.

4) Чтобы запускать контейнеры через 1 compose файл так, чтобы они "не видели друг друга" достаточно настроить сети networks, привязать отдельные контейнеры к нужным сетям.
Изоляция построена на основе технологии network namespaces, которая создаёт каждому контейнеру свой отдельный сетевой стек, со своим айпи и маршрутизацией. Внутри одной сети контейнеры могут связываться друг с другом через днс имена
